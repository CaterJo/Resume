## Core value

### 1.프론트엔드는 어떤 역할이라 생각하는지?

> 고객에게 물 흐르듯 자연스러운 경험을 제공하며 이탈을 최소화는 것

고객이 제품을 만나는 첫 순간이 프론트엔드입니다.  
그렇기 때문에 프론트엔드 개발자에게 가장 중요한 것은 엔드 유저의 경험을 공감하는 것이라 생각합니다.  
사용자 관점에서 자연스러운 사고 흐름에 기반한 기능을 제공하는 것과 부드러운 인터렉션을 경험하게 하는 것을 가장 중요하게 생각합니다.

제가 생각하는 프론트 엔드의 역할은 서비스를 이용하는 사용자의 마음에 공감하고 물흐르듯 자연스러운 경험을 제공하는 것입니다.

### 2.업무에서 중요하게 생각하는 것이 있다면?

- 팀워크를 최우선합니다.

  - **개인의 퍼포먼스**보다 중요한 것은 **팀 퍼포먼스**라 생각합니다.

- 동료의 페인 포인트를 기술과 시스템 개선으로 해결합니다.

  - _"PR 가독성이 떨어져요"_ → PR 템플릿 도입, gif or 동영상 첨부
  - _"redux에서 서버 데이터를 처리하기 번거로워요"_ → reat-query
  - _"공통 컴포넌트 여부를 알기 어렵고 사용방법을 이해하기 어려워요"_ → storybook
  - _"실패 케이스와 성공 케이스가 혼재되서 코드가 복잡해요"_ → Errorboundary

- 동료의 시간을 아껴주는 것을 중요하게 생각합니다.

  - 문서화로 같은 시행착오가 반복되는 것을 막습니다.
  - 빈번하게 반복되는 커뮤니케이션을 템플릿으로 해결합니다 (지라, github pr)
  - util함수처럼 사용빈도가 높은 코드는 단위테스트를 거쳐 배포합니다.

- 간결하고 명확한 커뮤니케이션을 추구합니다.
  - 텍스트 기반 커뮤니케이션시 이미지 첨부
  - 기술이나 코드 변경을 제안할 경우 판단에 참고할 레퍼런스 첨부
- 유지보수를 최소화할 수 있는 코드를 생산합니다.
  - 누구든 유지보수할 수 있는 코드를 목표합니다.
  - 코드에 대한 오너십을 분산시킵니다.

### 3.기술 선정에서 중요하게 생각하는 것은?

1. 기술을 통해 해결하고자 하는 문제점이 무엇인지?
2. 직접 구현했을 때의 문제점이 있다면?
3. 기획 요구사항을 충족하는지?

6. 국내 채용시장의 사용자풀이 확보됐는지? (유지보수 관점)

### 4.코드리뷰시 중요하게 생각하는 가치는?

1. 작성자의 의도를 이해하고 **존중**하는 것.
2. 리뷰어가 선한 마음으로 더 나은 결과를 위해 리뷰를 했다는 믿음을 갖는 것.
3. 내 생각이 틀렸을 수 있다는 수용의 여지를 남기는 것.
4. 레퍼런스를 통해 객관적 판단 근거를 제공하는 것.
5. 주관적이고 모호한 표현을 지양하며 구체적 개선 포인트, 리스크 요인을 설명하는 것.

   _"이 props이 이 컴포넌트에서 어떤 역할을 수행하는지 애매하네요. 이렇게 이름을 바꿔보면 어떨까요?"_  
   _"의존성 배열에 이 요소 때문에 effect함수가 불필요하게 실행될 것 같은데 이렇게 바꿔보면 어떨까요?"_

### 5.개발에서 중요하게 여기는 가치가 있다면?

> 유지보수하기 쉬운 코드  
> 가독성 좋은 코드  
> 건강한 부채를 가진 코드  
> 오래 `생존`할 수 있는 코드

제가 가장 중요하게 생각하는 것은 `오래 생존할 수 있는 코드`를 생산하는 것입니다.  
그러기 위한 전제 조건은 `기업의 생존`입니다. 이런 가치를 중요시하기 때문에 코드 레이어를 벗어나 비지니스 문제를 먼저 고민하고 생존을 위한 기술적 부채를 적절히 수용합니다.  
그 다음으로 중요한 것이 `유지보수`입니다. `유지보수`하기 쉬우려면 `가독성`이 좋아야합니다.  
`가독성 좋다`는 것은 새로운 요구사항에 빠르고 유연하게 대처할 수 있다는 것을 의미합니다.  
정리하자면 **가독성이 좋은 코드가 변화에 적응하며 시장에서 생존할 수 있는 코드**라 생각합니다.

- 동료와 영감을 주고받는것.

### 7.개발 문화 개선을 위해 행동한 것이 있다면?

**코드 컨벤션 도입**
